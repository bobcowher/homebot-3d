# HomeBot-3D ‚Äî Realistic House Redesign

**Date:** 2026-07-03
**Status:** Design ‚Äî awaiting review

## Context

The nav spine, embodiment, and POV teleop are merged to `main`. Driving the
sim surfaced that the world's fidelity is too low for the intended use: walls
are a full 0.5 m thick (a filled grid tile), doorways are 0.5 m wide (7 cm of
clearance for the robot, which catches and stalls), surfaces are flat-shaded
primitives, and the observation is 84√ó84. The goal now is a **fairly realistic
house that makes sim-to-real transfer simple-ish**, without abandoning the
project's constraints (single-env, deterministic, no massive parallelism, no
heavy third-party dependencies ‚Äî see [[robert-research-world-models]]).

## Decisions (locked in brainstorming)

- **Transfer target:** generic **Bumperbot-class** differential-drive robot
  (two drive wheels + caster + camera). Kept parametric; will be re-dimensioned
  to specific hardware later.
- **Observation resolution:** **128√ó128 RGB** egocentric (up from 84√ó84).
- **Realism approach:** **textured procedural** ‚Äî procedural geometry dressed
  with **procedurally generated, in-repo texture PNGs** (no downloads) + soft
  indoor lighting. Meshes are a possible future upgrade, out of scope here.
- **Furnishing:** **lightly furnished** ‚Äî a few believable pieces per room plus
  the goal fixtures.
- **Floorplan:** keep the existing 4-room layout (living / kitchen / hall /
  bedroom), re-proportioned; widen doorways.
- **Domain randomization:** **deferred** ‚Äî documented as a future seam, not
  built now.

## Goals

- Walls read as real walls (thin), doorways are comfortably passable by the
  robot, and the robot no longer catches/stalls on wall geometry.
- Surfaces (floor, walls, furniture) carry believable textures at 128 px.
- Each room is lightly furnished with textured primitives that double as
  navigation obstacles.
- Robot proportions/camera reflect a plausible Bumperbot-class platform;
  everything dimensional lives in `constants.py` for easy re-speccing.
- Determinism preserved; no new third-party dependencies (textures are static
  bytes generated by an in-repo script); single-env focus intact.

## Non-Goals

- No mesh/URDF import (textured primitives only this phase).
- No domain randomization implementation (future seam, documented only).
- No change to the diff-drive action interface or the privileged seam.
- No new maps beyond the re-proportioned default house.

## Component 1 ‚Äî Thin walls & doorways (`world.py`, `maps.py`)

Replace filled-cell wall boxes with **boundary-extracted thin panels**
(validated in prototype): for each floor cell, emit a thin wall panel
(`WALL_THICK ‚âà 0.08 m`) on each edge shared with a wall cell or the grid
border; panels overlap slightly on their long axis to close corner gaps. All
panels named `wall_{i}` so `Robot.collided()` still detects them.

`maps.py`: widen the room-connecting doorways in `DefaultHouseMap` to **2 tiles**
and re-check room dimensions so furniture leaves navigable paths. Add simple
door **frames** (short thick posts flanking each opening) for realism.

Prototype result to preserve: robot drives straight through a doorway with >1 m
of clear progress (no stall).

## Component 2 ‚Äî Texture system (`scripts/gen_textures.py`, `homebot3d/assets/`, `world.py`)

- **Generator** `scripts/gen_textures.py`: a seeded NumPy script producing
  texture PNGs ‚Äî `floor_wood.png`, `floor_tile.png`, `wall_paint.png`,
  `wood.png`, `fabric.png` ‚Äî written to `homebot3d/assets/textures/`. Run once;
  the PNGs are committed (deterministic static bytes). Uses the same stdlib PNG
  writer pattern already in `scripts/render_dev.py`.
- **Loading:** `world.texture_assets() -> dict[str, bytes]` reads the committed
  PNGs. The MJCF `<asset>` block references them by filename
  (`<texture type="2d" file="floor_wood.png"/>` + `<material>` wrappers).
- **Compilation seam:** add `world.compile_model(map, robot_start=None) ->
  mujoco.MjModel` that calls
  `mujoco.MjModel.from_xml_string(build_mjcf(...), texture_assets())`. All model
  builders (env, tests, `render_dev.py`, `play.py`) switch to `compile_model`
  so texture bytes are always supplied without filesystem-path fragility.

## Component 3 ‚Äî Furniture (`maps.py`, `world.py`)

- `maps.py`: add a `furniture` list to `DefaultHouseMap` ‚Äî entries of
  `(kind, col, row)` for non-goal pieces: `sofa`, `coffee_table` (living);
  `bed`, `nightstand` (bedroom); `counter`, `kitchen_table` (kitchen).
- `world.py`: a `_furniture_piece(kind, cx, cy)` builder emits textured
  multi-geom primitives per kind, all geoms `fixture_`-prefixed (collision) and
  x/y half-extents bounded so placed pieces leave navigable paths. Goal fixtures
  (fridge/recliner/door) are reshaped/textured through the same path.
- Placement is validated: robot start and all goal targets remain reachable
  (existing start-sampling + integration drive + reach tests).

## Component 4 ‚Äî Robot & camera (`world.py`, `constants.py`)

- Tidy the primitive robot toward Bumperbot proportions (two drive wheels +
  caster, camera on a short mast). Cosmetic geoms stay massless/non-colliding;
  `robot_body` collision cylinder and the drive servo unchanged.
- Camera: set a realistic **`fovy ‚âà 75¬∞`** on the `ego` camera; camera height
  `CAMERA_HEIGHT` set for a small robot (~0.5 m). All in `constants.py`.

## Component 5 ‚Äî Observation resolution (`env.py`, `constants.py`)

- Default obs `width = height = 128`; update `observation_space` (both
  `HomeBot3DEnv` and the `HomeBot3DGoalEnv` Dict). Existing tests asserting
  `(84, 84, 3)` update to `(128, 128, 3)`.
- Privileged seam and action interface unchanged.

## Determinism & sim-to-real notes

- Textures are static committed bytes; lighting/materials are fixed; RNG paths
  unchanged ‚Üí determinism test stays green.
- Sim-to-real is served by realistic metric scale (have it), realistic
  appearance (textures + soft lighting), and a plausible camera (height/FOV/res).
  A per-episode texture/light **domain-randomization hook** is a documented
  future seam, not built here.

## Testing strategy

- **Walls:** panels are thin (`geom_size` ‚âà `WALL_THICK/2` on the thin axis),
  all `wall_`-named; a doorway drive-through test asserts the robot clears the
  opening (no stall) ‚Äî the concrete regression for the reported bug.
- **Textures:** `texture_assets()` returns the expected filenames as non-empty
  bytes; `compile_model` builds a stepping model with materials assigned.
- **Furniture:** each piece present, all geoms `fixture_`-prefixed, reach-bound
  respected; integration drive still scores.
- **Robot/camera:** ego camera `fovy` set; camera world height ‚âà `CAMERA_HEIGHT`.
- **Obs:** reset/step obs shape `(128, 128, 3)`; determinism test green.
- **Visual:** `render_dev.py` at 128 px ‚Äî inspect ego + overview frames for
  thin walls, textures, furniture, doorway realism.
- Interactive teleop (`play.py`) re-verified by the user at the end.

## Rollout (one phased plan)

thin walls & doorways ‚Üí texture generation + compile seam ‚Üí furniture ‚Üí robot &
camera ‚Üí obs resolution. Built with subagent-driven TDD + review, same as the
prior feature.
